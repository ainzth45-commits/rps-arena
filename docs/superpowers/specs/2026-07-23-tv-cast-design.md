# สตรีมเกมขึ้นจอ TV — Design (2026-07-23)

เกมที่ 3 "เป่า ยิ้ง ฉุบ! อารีน่า!" · เจ้านายเคาะแนวทางแล้วก่อนลงมือ

---

## 1. เป้าหมาย

ให้ทั้งทีม (~12 คน) ดูเกมร่วมกันบนจอ TV 55" ระหว่างที่คนคุมเล่นบน iPad
แรงบันดาลใจ: Kahoot — ผู้เล่นทำบนมือถือ ผลโชว์จอใหญ่ ทุกคนลุ้นพร้อมกัน

- **ปกติ** TV โชว์หน้าอันดับ + move rate (ตามกติกาภาษีของแชมป์)
- **ตอนดวล** TV สตรีมตามทุกจังหวะ (ปะทะ VS → เป่ายิ้งฉุบ → ผล) พร้อมเสียง
- **ดวลจบ** TV กลับหน้าอันดับ
- จับคู่ **เฉพาะ iPad + TV คู่นั้น** ด้วยรหัสห้อง เครื่องอื่นไม่ sync

---

## 2. หลักการที่ห้ามพลาด (รากฐาน)

🔴 **เกมบน iPad ต้องเล่นได้ครบ 100% ต่อให้ไม่มีเน็ต / Supabase ล่ม / ไม่ได้เชื่อม TV**

การสตรีมขึ้น TV เป็น **ของแถม best-effort** — ยิงข้อความแบบ try/catch ถ้าส่งไม่ได้ก็เงียบ
เกมเดินต่อปกติ ไม่มีวันที่ TV ทำเกมค้างหรือช้าลง

---

## 3. สถาปัตยกรรม

- **Supabase Realtime Broadcast ล้วน — ไม่มีตาราง ไม่มี RLS** (ส่งข้อความสดผ่านห้อง)
- **ห้อง** = ชื่อช่องตามรหัส เช่น `arena-4821` · TV subscribe · iPad broadcast
- **Presence** ของ Supabase บอกว่า TV เข้าห้องเมื่อไหร่ → iPad ส่ง snapshot เต็มให้ทันที
  (Broadcast ไม่เก็บย้อนหลัง TV เข้าตอนไหนก็ต้องได้ข้อมูลล่าสุดเสมอ)
- **เกมกับ TV = แอปเดียวกัน คนละหน้า:**
  - `.../rps-arena/` = เกม (iPad)
  - `.../rps-arena/?tv` = จอ display (TV)
- **Deploy เหมือนเดิมทุกอย่าง** — GitHub Pages static · `git push` แล้วขึ้นทั้งคู่พร้อมกัน
- ใช้ **anon key เท่านั้น** (publishable ปลอดภัยกับ repo public) · ห้ามแตะ service_role key
  เพราะไม่มีตาราง จึงไม่มีข้อมูลให้ดึงผ่าน key ได้ — แค่ส่งข้อความสดผ่านห้องที่ต้องรู้รหัส

## 4. โครงโค้ด (แยกส่วนชัด ทดสอบได้)

| ไฟล์ใหม่ | หน้าที่ | พึ่งพา |
|---|---|---|
| `src/tv/tvChannel.ts` | คุยกับ Supabase (connect/broadcast/subscribe/presence) · ห่อ try/catch ทั้งหมด · best-effort | supabase-js |
| `src/tv/tvView.ts` | **ฟังก์ชันบริสุทธิ์** แปลง game state → `TvView` · เทสได้ไม่ต้องมีเน็ต | domain เดิม |
| `src/tv/roomCode.ts` | สุ่ม/ตรวจ/normalize รหัสห้อง · เทสได้ | — |
| `src/tv/TvDisplayApp.tsx` | แอปฝั่ง TV — เริ่มด้วย `BootScene` (โหลดรูป+ปลุกเสียง) → จับคู่ → รับ `TvView` มาเรนเดอร์ (ใช้ scene + preload เดิมซ้ำ) | BootScene · scene · preloadAssets เดิม |
| `src/tv/PairScene.tsx` | หน้าจับคู่: ฝั่ง iPad พิมพ์รหัส · ฝั่ง TV โชว์รหัส + "แตะเพื่อเริ่ม" | — |

**main.tsx:** เช็ค `?tv` ใน URL → เรนเดอร์ `<TvDisplayApp/>` แทน `<App/>`

## 5. ข้อมูลที่ส่งขึ้น TV (TvView projections)

`TvView` เป็น union ตาม `kind` — iPad ส่งเป็นข้อความเดียวต่อการเปลี่ยน state:

| kind | ส่งเมื่อ | ข้อมูล |
|---|---|---|
| `leaderboard` | อยู่หน้า home/อันดับ · หรือส่ง snapshot ตอน TV เข้าห้อง | top 10 (id/ชื่อ/รูป/คะแนน) + move rate เฉพาะ top 3 (ตาม `visibleMoveRates`) + season id |
| `versus` | เข้าฉากปะทะ | คู่ A/B (รูป/ชื่อ/อันดับ/สตรีค) + สถิติเจอกัน + สุ่มหรือเลือก |
| `movePick` | เข้าหน้าเลือกมูฟ | คู่ A/B + **เวลาเส้นตาย (timestamp)** ให้ TV นับเอง + เลือกยัง (boolean · ไม่บอกมูฟ) |
| `shoot` | เปิดมูฟพร้อมกัน | มูฟทั้งสองฝั่ง + ผล (reveal) |
| `result` | จอผล | ผู้ชนะ/แพ้ + มูฟ + คะแนนที่ได้เสีย + สตรีค |
| `seasonEnd` | ประกาศแชมป์ | โพเดียม top 3 |

**timer ซิงค์ด้วยเส้นตาย:** ส่ง deadline ครั้งเดียว → TV นับถอยหลังเอง ตรงกับ iPad โดยไม่ต้องส่งทุกวินาที

## 6. สถานะฝั่ง TV (4 สถานะ)

| สถานะ | เมื่อไหร่ | TV โชว์ |
|---|---|---|
| **0. หน้าเปิด (boot)** | เปิด `?tv` ทุกครั้ง | **ใช้ `BootScene` ตัวเดียวกับเกม** — โลโก้ + "แตะเพื่อเริ่ม" → แตะแล้ว **โหลดรูปทั้งเกมให้ครบ** (หน้าโหลด % เดิม) + **ปลุกเสียง** (การแตะ = user gesture) → เข้าสถานะถัดไป |
| **1. ยังไม่เชื่อม** | โหลดเสร็จ ยังไม่มี iPad จับคู่ | หน้าจับคู่: รหัสตัวใหญ่ + "รอ iPad เชื่อม..." |
| **2. เชื่อม · iPad เล่นอยู่** | กำลังเล่นปกติ | หน้าอันดับ / สตรีมดวล ตาม TvView |
| **3. เชื่อม · iPad ปิด/พัก** | iPad ไม่ได้เล่น | อันดับล่าสุดที่เคยรับ (แช่ไว้) + จุดเล็ก "iPad ไม่ได้เชื่อมอยู่" |

**หมายเหตุ:** สถานะ 0 ใช้ `BootScene` + `preloadAllGameAssets` ที่มีอยู่แล้วซ้ำ — ไม่เขียนใหม่ · แตะโลโก้ = ปลุก `AudioContext` (เหมือนเกม) จึงไม่ต้องมีปุ่ม "แตะเพื่อเริ่ม" แยกอีก

- **TV cache อันดับล่าสุดใน localStorage** → รีเฟรช TV เห็นอันดับทันที ไม่ต้องรอ iPad
- **TV จำรหัสตัวเองใน localStorage** → รีเฟรชได้รหัสเดิม
- **iPad จำการเชื่อมใน localStorage** → เปิดเกมครั้งหน้าเชื่อมอัตโนมัติ (มีปุ่มเลิกเชื่อม)

---

## 7. ดวลนอกรอบ (กันสปอยล์)

ระหว่างสองคนเลือกมูฟ (ผลัดกันถือ iPad) → iPad ส่งแค่ view กลางๆ **"กำลังดวลนอกรอบ..."** (ไม่มีข้อมูลมูฟ)
**หลังทั้งคู่เลือกครบ**ค่อยส่ง `versus` → `shoot` → `result` ตามปกติ

---

## 8. เสียงบน TV

TV เล่นเสียงเองด้วย `sfx.ts` ตัวเดิม (trigger ตอน TvView เปลี่ยนเข้าฉากที่มีเสียง)
เบราว์เซอร์ต้องมีการแตะครั้งแรกก่อนเล่นเสียง → **การแตะโลโก้ในหน้า boot (สถานะ 0) ปลุก AudioContext ให้แล้ว**
(ใช้ `BootScene` + `unlockAudio()` ตัวเดียวกับเกม ไม่ต้องมีปุ่มแยก)

---

## 9. การจับคู่ (flow)

**นิยามรหัส:** รหัสจริง = ตัวเลข 4 หลัก (เช่น `4821`) · TV โชว์เป็น `ARENA-4821` เพื่ออ่านง่าย
แต่ iPad พิมพ์แค่ `4821` · ช่องจริงใน Supabase = `arena-4821` (ประกอบจาก prefix + เลข)
`roomCode.ts` normalize input: ตัดช่องว่าง/ตัด prefix "ARENA-" ถ้าเผลอพิมพ์ติดมา/รับได้ทั้ง 4821

0. **TV** เปิด `?tv` → หน้า boot (โลโก้) → **แตะโลโก้ → โหลดรูปครบ + ปลุกเสียง**
1. **TV** สุ่มเลข 4 หลัก (จำใน localStorage) → โชว์ตัวใหญ่ `ARENA-4821`
2. **iPad** ตั้งค่า → "เชื่อมจอ TV" → พิมพ์ `4821` → เชื่อมห้อง `arena-4821` (จำใน localStorage)
3. iPad ส่ง snapshot อันดับล่าสุดทันที · TV เปลี่ยนเป็นหน้าอันดับ
4. เปิดเกมครั้งหน้า iPad เชื่อมห้องเดิมอัตโนมัติ

## 10. การทดสอบ

- **Unit test (ไม่ต้องมีเน็ต):**
  - `tvView.ts` — game state + phase → TvView ถูกต้องทุก kind · ดวลนอกรอบไม่รั่วมูฟก่อนเวลา
  - `roomCode.ts` — สุ่มไม่ซ้ำรูปแบบ · normalize (ตัดช่องว่าง/พิมพ์ใหญ่) · ตรวจรหัสผิด
- **Browser test:** เปิด 2 แท็บ (เกม + `?tv`) เชื่อมจริง เดินดวล ดูว่า TV ตามครบทุกฉาก + เสียงออก
- **Offline test:** ตัดเน็ต → เกม iPad ยังเล่นได้ครบทุกจอ (พิสูจน์หลักการข้อ 2) · TV ขึ้น "เชื่อมไม่ได้" ไม่ค้าง

## 11. สิ่งที่ไม่ทำในเฟสนี้ (YAGNI)

- ไม่ทำ 2 iPad + 1 TV (คู่เดียวพอ)
- ไม่ทำ replay/บันทึกวิดีโอ
- ไม่ทำ interactive บน TV (TV เป็นจอดูอย่างเดียว)
- ไม่ทำ leaderboard แบบ real-time หลายห้องพร้อมกัน

## 12. ความเสี่ยงที่รู้ตัว

- **เน็ตออฟฟิศหลุด** → เกมยังเล่นได้ (best-effort) · TV แช่หน้าเดิม พอเน็ตกลับมาต่อเอง
- **Supabase free tier limit** → เป็นเครื่องมือในทีม ปริมาณข้อความน้อยมาก (ส่งเป็นครั้งๆ ไม่ใช่ทุกเฟรม) ไม่น่าถึงลิมิต
- **anon key เปิดเผยใน repo public** → ปลอดภัยโดยดีไซน์ (ไม่มีตาราง ไม่มีข้อมูลให้ดึง)
- **ต้องมี Supabase project + anon key จากเจ้านาย** ก่อน implement (เจ้านายจะสร้าง project ใหม่)
